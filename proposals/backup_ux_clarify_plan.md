# 备份功能现状诊断与可理解性改进规划

> 目标：回答「备份生效吗 / 目录文件大小在哪看 / 规则是按天还是什么」，并规划如何让它可见、可懂、可控。
> 范围：仅改进备份的**可观测与可理解性**，不改动现有备份/回滚核心逻辑（安全模型不变）。

## 一、现状诊断

### 1. 是否生效？—— 生效，且已闭环
- 后端 `backup.rs::backup_before_overwrite` 在以下操作**改写/删除已有文件前**自动拷贝原文件：
  `write_files`(覆盖时)、`edit_files`、`delete_files`、`copy_files`(覆盖时)、`move_files`(覆盖时)。
- 已具备**一键回滚**（`restore_file`）与**变更 diff**（`get_file_diff`），在审计日志页（LogTab）每条写操作详情里可还原/查看——备份链已闭环。

### 2. 目录 / 文件 / 大小 当前情况（用户痛点来源）
- **目录**：`{app_data_dir}/.cc-bridge-backup/` → Windows 约
  `C:\Users\{用户}\AppData\Roaming\{应用标识}\.cc-bridge-backup\`。
  设置页只显示**相对路径字符串** `.cc-bridge-backup`（不显示绝对路径）；
  「浏览…」按钮是**选择目录来改配置**，不是打开已有备份文件夹。→ 用户找不到、看不懂。
- **文件**：所有被改写的原文件混在同一文件夹，命名 `{原文件名}.{YYYYMMDD_HHMMSS_mmm}.bak`，按原文件名前缀归类。UI 无任何清单。→ 用户不知道里面有哪些文件。
- **大小**：**任何地方都不显示**（无单文件大小、无总占用）。→ 用户不知道占多少空间。

### 3. 规则：按天？还是什么？—— 不是按天
是**「每次改写/删除已有文件前的操作快照」**：
- 触发：某次操作即将覆盖/删除一个**已存在**的文件（且「写操作自动备份」开关开启）。
- 动作：拷贝该文件当前内容为 1 个带毫秒时间戳的 `.bak`。
- 保留：`prune_backups` 对**同一原文件名**只保留**最近 N 份**（默认 10），超出删最旧。
- 即：**按文件计版本，随编辑次数累积，与日期无关**。一个被改 50 次的文件留最近 10 版；只改过 1 次的文件只有 1 个备份。无时间维度轮换。
- 用户困惑点：以为「按天保留」，实际是「按文件保留最近 N 版」。

## 二、改进规划

### P0（高价值 · 低风险 · 纯前端 + 1 个小后端命令）
- **A. 说清规则**：备份卡片文案改为人话——「每次改写/删除已有文件前自动留一份快照；同一文件最多保留最近 N 份（非按天）。」
- **B. 目录可直达**：显示**解析后的绝对路径**（不是相对字符串）；新增「打开备份文件夹」按钮（复用 install_dir 的 `explorer /select` 模式），用户可直接看到 `.bak` 文件。
- **C. 显示大小**：后端新增 `backup_stats` 命令（遍历备份目录，返回 文件数 / 总字节 / 最早·最晚时间），设置页展示「共 N 个备份 · 占用 X.X MB」。

### P1（中价值）
- **D. 备份清单**：备份卡片下加可折叠「查看备份文件」，按原文件名分组列出 `.bak`（时间 + 大小），每条可「打开 / 还原」（接已有 `restore_file`）。直接回答「备份了哪些文件」。

### P2（可选 · 谨慎）
- **E. 规则可配**：是否加「按天 / 保留天数」维度？当前按文件版本其实更安全（保证任意文件总有近 N 版）。建议**保留现状规则、只把文案讲清**，不引时间维度以免误删。如确需，可加「保留天数」作为**额外**清理条件（与份数取 AND）。

## 三、实施顺序建议
1. 先 P0（A 文案 + B 打开文件夹 + C 大小统计），约 1 个后端命令 + 设置页 `SecurityTab` 备份区改动。
2. 再 P1（D 备份清单弹窗）。
3. 不碰 `backup.rs` / 回滚 / diff 核心逻辑（安全模块不削弱）。

## 四、验收口径
- 用户在设置页能：看懂规则一句话、点开看到真实备份文件夹、看到总占用与份数。
- 可选：展开看到「哪个原文件有哪几份备份、各多大、何时」。
- 现有回滚 / diff 行为不受影响。
