## High（建议优先处理）

### H1. ProjectSteps 复制按钮把 cd 前缀丢掉（双重写剪贴板，第二次覆盖第一次）
`desktop/src/components/tabs/connect/Steps.tsx:62` · fe-connect · correctness

- 问题：handleProjectCopy 先 writeText(fullCommand=`cd <path> && <connectCommand>`) 再调 onCopy()，而 onCopy=handleCopy 内部又 copyText(connectCommand)（不含 cd）。两次写剪贴板背靠背，后者覆盖前者，最终剪贴板是不带 cd 的命令。
- 后果：项目级作用域下点复制，UI 显示已复制，但粘贴到远程执行的命令缺 `cd <项目路径> &&`，.mcp.json 写到错误目录，之后连接静默失败。ProjectSteps「自动合并成一条命令」的卖点被抵消。
- 修复：ProjectSteps 复制走单一路径，把 fullCommand 上抛由 handleCopy 用 copyText(fullCommand) 一次写入；或 ProjectSteps 完全自管复制，不复用会重写 connectCommand 的 handleCopy。

### H2. SSE tools/call 未进入 with_op_backup 作用域，写工具会 panic
`desktop/src-tauri/src/mcp/sse.rs:169` · mcp-transport · correctness/panic

- 问题：SSE 的 tools/call 直接 dispatch_tool，没像 http.rs/batch.rs 那样用 with_op_backup 包裹。写工具备份后调 record_op_backup → OP_BACKUP.with()（task_local，未建作用域即 panic，与 timing 用的 try_with 不同）。
- 后果：非只读下经 SSE 对已存在且 backup_enabled 的文件调 write/edit/delete/move/copy → panic；连接被重置、不返错、备份已生成但写入未执行、无审计。edit_files 恒操作已存在文件，经 SSE 编辑必 panic。
- 修复：SSE 的 tools/call 用 with_op_backup 包裹 dispatch_tool（与 HTTP 一致）；或 record_op_backup 改 try_with 降级。

### H3. SSE 传输通道完全绕过审计日志
`desktop/src-tauri/src/mcp/sse.rs:169` · mcp-transport · security/audit-bypass

- 问题：SSE 的 tools/call 只做 increment_requests/record_request_time，全程不调 write_audit_for_call。HTTP 通道每次调用都写审计。
- 后果：经 SSE 调 run_command 执行任意命令、或 read_files 读白名单内敏感文件，实际执行却在 audit.log 不留痕迹（仅 batch 会自审子操作）。与「所有操作受审计」的核心卖点冲突。
- 修复：SSE 路径复用 write_audit_for_call（带 source_ip），与 HTTP 一致。

### H4. 后台命令 wait 线程整程持锁，stop_command 无法终止运行中进程
`desktop/src-tauri/src/mcp/tools/run_command.rs:503` · run-command · concurrency/deadlock

- 问题：spawn_background 的 wait 线程 `child.lock().unwrap(); c.wait()`——StdMutex 守卫在阻塞 wait() 全程持有（进程退出才释放）。stop_command 需对同一 Arc<StdMutex<child>> 执行 start_kill()，两者争同一把锁。
- 后果：起一个失控后台命令（如 ping -t）再 stop_command，child.lock() 一直阻塞到进程自己退出——失控进程永远杀不掉，且 async 里的 std 阻塞锁把一个 tokio 工作线程占死。恰好摧毁 stop_command 的存在意义。
- 修复：wait 线程不要在 wait() 期间持锁（try_wait 轮询短持锁），或用 tokio::process 让 kill 与 wait 不争独占锁。

### H5. content 搜索 counter 累加共享 Vec 总长，导致提前截断、结果不完整
`desktop/src-tauri/src/mcp/tools/search_files.rs:278` · search-files · correctness

- 问题：content 分支 `let n = g.len(); counter.fetch_add(n)`——n 是共享累加器 matches 的总长度而非本文件新增数（name 分支正确地 +1）。counter 被前缀和式放大。
- 后果：head_limit=100 时，实际仅约 45 条结果就提前触发 Quit 并置 truncated。内容搜索静默丢失大量命中（可达 30%~50%）且误报「已截断」。核心检索工具的正确性缺陷。
- 修复：counter 只加本文件真正 push 的条数；截断判定仍用 g.len()>=head_limit，把「总长」与「增量」两个语义分开。

### H6. 内容搜索绕过扩展名白名单，可逐行读出任意扩展名文件内容
`desktop/src-tauri/src/mcp/tools/search_files.rs:78` · search-files · security

- 问题：handle 只对 rootPath 校验白名单根，遍历到的每个文件不做扩展名白名单（allowedExtensions）校验。content 模式返回体直接带命中行原文与上下文行。read_files 受扩展名白名单约束，search_files 内容搜索不受。
- 后果：对某 allowed root 发起 contentPattern="."、outputMode=content，即可把根目录下 .env/.pem/id_rsa/*.key 等不在白名单的敏感文件逐行回显，绕过扩展名访问控制，信息泄露。
- 修复：遍历命中后、返回内容前，对每个文件应用与 read_files 一致的扩展名白名单校验。

## Medium（23 条，摘要）

1. **export_config 明文导出 Bearer token**（commands.rs:1409）——导出的配置 JSON 含唯一远程凭证明文，发出/入库即泄露全部访问权限。导出时脱敏 token。
2. **prune_backups 用文件名前缀匹配跨文件误删备份**（backup.rs:63）——同名不同目录、或前缀包含（config / config.local）会误删他文件备份。应按 original_path 精确筛选。
3. **UTF-16 文件 BOM 丢失且写回必然失败**（encoding.rs:76）——had_bom 只覆盖 UTF-8；encoding_rs 把 UTF-16 编码降级为 UTF-8，任何写回触发 round-trip 报错。UTF-16 可读不可写。
4. **handleProjectCopy 未 await/catch 剪贴板**（Steps.tsx:63）——H6 修复在此处回归，权限被拒时静默 reject。
5. **展开行用数组下标做 key/状态，轮询后详情张冠李戴**（LogTab.tsx:335）——10s 轮询插入新记录后 expandedRow 指向另一条，点「一键还原」可能还原错误条目、覆盖非预期文件。用稳定 id。
6. **CSV 导出公式注入**（LogTab.tsx:106）——params 等不可信字段以 =/+/-/@ 开头未加前缀，运维用 Excel 打开审计表时公式执行。导出前加防注入前缀。
7. **性能诊断只统计当前页却标称「基于全部」**（PerfCharts.tsx:199）——P95/最慢N次只是当前页极值，误判瓶颈。改为全量或改文案。
8. **buildPermissionGrantCommand 非-shell 授权命令被 shell 截断失效**（utils.ts:205）——JSON.stringify 的内层双引号截断 python3 -c "..."。「不含命令执行工具」的更安全默认路径整段不可用。用单引号构造 python 列表。
9. **write_files 非原子覆写**（write_files.rs:110）——直接 tokio::fs::write，崩溃/断电会截断损坏原文件；edit_files/notebook 已用原子写。复用 write_atomic。
10. **notebook_edit 缺文件大小校验**（notebook_edit.rs:46）——不受 max_file_size_bytes 约束，超大 .ipynb OOM。补 assert_file_size_ok。
11. **notebook_edit 覆写前不备份、不写审计关联**（notebook_edit.rs:72）——同一文件用 edit_files 可回滚、用 notebook_edit 不可，安全承诺缺口。补 backup_before_overwrite。
12. **流式行范围读取按字节 0x0A 切分，UTF-16 override 乱码**（read_files.rs:197）——UTF-16 换行是 0A 00，单字节切会截断字符对。流式分支仅对 ASCII 兼容换行编码启用。
13. **remove_directory recursive 绕过备份与扩展名白名单**（remove_directory.rs:41）——remove_dir_all 不备份、不校验扩展名，等于绕过所有删除防护的后门。递归删前逐文件备份/校验。
14. **/health 与 /mcp 共用限流时间戳向量，限流可被绕过**（http.rs:208）——/health 的 1s retain 清掉 /mcp 长窗口的历史，每次 /mcp 前打个 /health 即约 60 倍绕过。两端点用独立限流存储。
15. **限流表清理是死代码，DashMap 无界增长**（http.rs:219）——push 后判 is_empty 恒 false，E-P2-1 从未生效，大量源 IP 使内存无界增长（DoS）。判空移到 retain 后、push 前。
16. **SSE 响应未按 SSE 帧封装，合规客户端收不到**（sse.rs:143）——裸 JSON 无 `data:` 前缀与 `\n\n`，SSE 解析器永不派发。整个 SSE 响应通道对合规客户端不可用。按 `event: message\ndata: ...\n\n` 封装。
17. **auth_middleware 对 /mcp/messages 同时跳过鉴权与限流**（http.rs:217）——SSE 工具调用不受限流；session 永不过期、经 URL query 传 token，等价长期能力令牌。加限流 + session TTL。
18. **SSE 会话注册表只增不删**（sse.rs:56）——连接断开无清理，反复建连泄漏 HashMap 条目 + 256 容量 channel。流结束时移除或后台 reaper。
19. **namePattern 仅对文件名匹配，含目录分量的 glob 静默失效**（search_files.rs:242）——`src/**/*.rs` 拿基名匹配永远不成立，返回空而非按路径过滤。对相对路径做 glob。
20. **IP 监听在 spawn_supervised 外多创建一个冗余 watcher**（main.rs:459）——临时 socket+线程建后即 drop，纯浪费。删除冗余代码。
21. **build_ip_sed_command 生成的 sed 命令语法损坏**（network.rs:104）——/mcp 放到分隔符外产生 5 个 #，托盘「复制 IP 替换命令」粘贴即报错，地址变动后远端无法重连。对齐前端把 /mcp 放进 pattern/replacement。
22. **parse_inbound_allow 按 \n\n 切块，CRLF 下失效**（firewall.rs:135）——netsh 输出是 CRLF，无连续 \n，整段当单块，端口放行检测恒 false，前端一直提示需开放防火墙（即使已放行）。规整化 CRLF 后再切。
23. **WSAIoctl 把任意 -1 当「地址已变化」**（ip_watch.rs:58）——未用 WSAGetLastError 区分错误，持续性错误下紧循环忙等 tx.send 占满 CPU。仅 WSAEFAULT 视为事件，其余退避/break。

## 观察与建议

1. **SSE 传输（2.3.8 新增）是重灾区**：6 条 high 里 3 条、多条 medium 都出自 `sse.rs`（panic、绕过审计、绕过限流、帧格式错误导致合规客户端根本收不到响应、会话泄漏）。这个功能目前既有安全缺口又可能对标准客户端不可用，建议在修复前不要作为推荐传输方式对外开放。
2. **安全模型的一致性缺口**是最系统的问题：多个工具/通道各自绕过了本应统一的防护（扩展名白名单、备份、审计、限流、只读拦截）——search_files/remove_directory/notebook_edit/SSE 各缺一环。建议把这些防护收敛为「所有工具调用入口的统一前置层」，而非每个工具各写一遍。
3. **前端命令拼接**多处 bug（H1 丢 cd、M8 授权命令截断、Low 的路径未加引号），这些是「拼错一个字符远程就连不上」的高风险纯函数，且零测试覆盖——对应我之前建议的「给 lib/utils.ts 加单元测试」。
4. 84 条中 55 条为 low（多为边界/健壮性/一致性），可作为长期清理项；high+medium 共 29 条建议优先排期。
