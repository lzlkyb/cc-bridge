# cc-bridge MCP 工具安全审查报告

> 审查日期：2026-07-12
> 审查对象：cc-bridge 桌面端 MCP HTTP Server 的全部 **17 个工具**
> 审查方式：逐文件亲审源码（注册 → 调度骨架 → 安全闸门 → 逐 handler 实现），非仅依赖文档描述
> 审查范围：`desktop/src-tauri/src/mcp/*`、`desktop/src-tauri/src/security/*`、`desktop/src-tauri/src/config.rs`

## 一、总评

**结论：17 个 MCP 工具的核心实现质量高，安全红线未被削弱，默认配置 fail-closed（默认安全）。未发现任何 P0 阻断级漏洞（无越权、无绕过安全闸门的路径）。可判定「核心没有问题」。**

但存在：
- **1 个 P1（重要）**：审计日志把完整 `arguments`（含 `write_files` 文件内容、`run_command` 命令字符串）序列化进 `audit.log`，存在信息泄露面。
- **3 个 P2（优化 / 一致性 / 已知局限）**：
  1. `notebook_edit` 写回直接 `tokio::fs::write`，未做原子写、未备份（其他破坏性写工具均有备份），一致性缺口。
  2. 危险命令拦截为启发式黑名单，代码注释自承可绕过（`rm -rf /home` 不命中，`echo "rm -rf /"` 误拦），属已知设计局限。
  3. `whitelist_enabled=false` 时路径白名单被整体跳过（仅 canonicalize），依赖 HTTP 层鉴权——需在文档/UI 明示这是「信任内网」模式。

> 说明：本次审查聚焦于「安全正确性 + 注册完整性 + 安全闸门是否被削弱」。功能性 / 性能细节不在本次范围，但审查中顺带确认了同步落盘审计已消除并发 NotFound 竞争、大文件流式读取、跨设备 move 回退等工程细节正确。

## 三、安全闸门逐项核验（全部正确）

### ① 路径白名单（最高频闸门）
- 文件：`security/path.rs::resolve_safe_path(requested, allowed_roots, enforce_whitelist)`
- 已存在路径 `canonicalize`；新建路径向上找已存在祖先再 `join`，避免 `../` 逃逸。
- 白名单比对用 `path.starts_with(root)`（**组件级**比对，非字符串前缀，无 `"/foo" 前缀命中 "/foobar"` 漏洞）。
- 越权时错误**附带 allowed_roots 白名单提示**（不泄露系统其他敏感路径）。
- 17 个文件/目录工具全部调用；`batch` 经 `dispatch_tool` 复用同一闸门。

### ② 鉴权（Bearer token）
- 文件：`security/auth.rs::verify_token`
- 用 `subtle::ConstantTimeEq` 做**常量时间比较**（长度不同直接 `false`，防时序侧信道）。
- `generate_token` 用 `rand` 32 位字母数字（~165 bit），足够。
- `auth_middleware` 在 `build_router` 中作为第一道中间件。

### ③ 限流键（防绕过）
- 文件：`mcp/http.rs::rate_limit_key`
- 限流键取自 **TCP 对端 IP**（真实连接源），**显式丢弃** `x-forwarded-for` 等可被伪造的 header（D1 修复，防 header 伪造绕过限流）。
- 限流开关 `rate_limit_enabled=true`（默认开）。

### ④ 扩展名 / 文件大小校验
- `security/extension.rs`：空白名单=全放行；大小写不敏感；无扩展名文件在白名单非空时拒。
- `security/filesize.rs::assert_file_size_ok`：校验磁盘文件大小，`max_file_size_bytes` 默认 20MB。
- 写工具同时校验「路径 + 扩展名 + 大小」三重。

### ⑤ 命令执行安全（最敏感）
- 文件：`mcp/tools/run_command.rs`
- `shell_enabled` **默认关**；开启后危险命令黑名单前置拦截（在解析 cwd/spawn 之前）。
- `process-wrap` JobObject + `CREATE_NO_WINDOW` / `CREATE_NEW_PROCESS_GROUP`，确保**整进程树被杀**（`TerminateJobObject`），无孤儿进程。
- 前台超时 30s、`max_output_bytes` 截断、后台并发上限 5。
- 测试覆盖 shell 关闭 / 危险命令 / cwd 越权 / 会话持久化。

### ⑥ 只读模式（READONLY）
- `dispatch_tool` 中 `WRITE_TOOLS` 数组（9 个具名写工具，**不含 batch**）。
- 已排除「batch 绕过只读」疑虑：`batch.rs` 通过 `Box::pin(dispatch_tool(...))` 路由每个子操作，**复用全部安检**（只读 / 白名单 / WRITE_TOOLS），测试 `readonly_blocks_write_in_batch` + `path_outside_root_rejected_in_batch` 覆盖。

### ⑦ 安全默认（fail-closed）
- `config.rs`：`whitelist_enabled=true`、`shell_enabled=false`、`rate_limit_enabled=true`、`max_file_size_bytes=20MB`、`allowed_roots=空`（空根=全拒，fail-closed）。

## 五、问题清单（按严重度）

### P1 — 审计日志全量序列化 `arguments`（重要）
- **位置**：`mcp/http.rs:376` `let args_str = arguments.to_string();` → `audit::new_entry(tool_name, &args_str, ...)`
- **问题**：`tools/call` 的完整参数（含 `write_files` 的**文件正文 base64/明文**、`run_command` 的**命令字符串**）被序列化进 `audit.log`。若 `max_file_size=20MB`，单次写入可能把近 20MB 内容落盘到审计日志；长期积累既是信息泄露面（日志常被宽松采集/转发），也是磁盘膨胀源。
- **建议**：审计仅记录「工具名 + 路径 + 操作类型 + 字节长度」，对 `write_files.content` / `run_command.command` 做脱敏或仅记长度。如需留痕，可改为「记录路径 + 哈希 + 大小」而非原文。

### P2-1 — `notebook_edit` 缺原子写 + 备份（一致性缺口）
- **位置**：`mcp/tools/notebook_edit.rs:99` `tokio::fs::write(&resolved, out)`
- **问题**：其他破坏性写工具（`edit_files`/`write_files`/`move_files`/`copy_files`/`delete_files`）均做**备份**，且 `edit_files` 用临时文件 + `rename` 原子写；而 `notebook_edit` 直接覆盖写回，无备份、无原子写。写入中途崩溃会损坏 `.ipynb`。
- **建议**：统一走「备份原文件 → 临时文件写 → rename 原子替换」模板（可抽公共 helper 到 `mcp/tools/`）。

### P2-2 — 危险命令黑名单为启发式兜底（已知局限）
- **位置**：`mcp/tools/run_command.rs:50-51` 注释自承
- **问题**：`rm -rf /home` 不命中、`echo "rm -rf /"` 误拦。仅最低成本兜底闸，不能替代沙箱。
- **建议**：二期升级为命令白名单 / 令牌化解析（D4 安全债）；当前依赖 `shell_enabled=false` 默认关闭作为主防线，可接受。

### P2-3 — `whitelist_enabled=false` 模式语义需明示
- **位置**：所有文件工具在 `whitelist_enabled=false` 时跳过 `resolve_safe_path` 的白名单比对（仅 canonicalize）。
- **问题**：此模式下路径越权保护依赖 HTTP 层 Bearer 鉴权，已非「白名单隔离」。若文档/UI 不提示，用户可能误以为仍受路径隔离保护。
- **建议**：在「设置页」与 README 明确：关闭白名单 = 「信任内网 / 仅鉴权」模式，不再做路径隔离。
