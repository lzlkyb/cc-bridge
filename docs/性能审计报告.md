# cc-bridge 全链路性能审计报告

> 审计范围：Rust 后端（HTTP 服务 / MCP 工具 / DB / 文件 IO / 审计写盘）+ React 前端（渲染 / 数据查询 / 轮询 / Bundle）+ Tauri IPC。  
> 日期：2026-07-12 | 版本基线：v2.2.21

## 总览

| 严重度 | 数量 | 说明 |
|--------|------|------|
| **P0 紧急** | 8 | 阻塞工作线程 / 无并发控制 / 反复 open 文件 / DB 无优化 |
| **P1 高** | 12 | 锁竞争 / 正则编译 / 事务缺失 / 整文件读内存 |
| **P2 中** | 13 | 内存泄漏 / 双重序列化 / 代码分割 / 组件重渲染 |
| **P3 低** | 6 | 体积优化 / 启动优化 / 外观 |

## 一、P0 紧急——阻塞 tokio 工作线程 / 无并发保护

### 1.1 notebook_edit：同步 std::fs 阻塞 async 运行时

**文件**: `desktop/src-tauri/src/mcp/tools/notebook_edit.rs` (L45, L97)
> `std::fs::read_to_string` / `std::fs::write` — 整个工具栈中唯一使用同步 I/O 的地方。文件在 NAS/HDD 上可阻塞 tokio 工作线程数秒。

**建议**: 替换为 `tokio::fs::read_to_string` / `tokio::fs::write`（2 行改动）。

### 1.2 list_allowed_roots：同步 std::fs 在 filter_map 闭包中

**文件**: `desktop/src-tauri/src/mcp/tools/list_allowed_roots.rs` (L28-47)
> `std::fs::metadata` + `std::fs::read_to_string` 在 async handler 中同步执行。任一 allowed_root 指向网络挂载路径可阻塞工作线程数秒。

**建议**: `tokio::fs::metadata` + `tokio::fs::read_to_string`，或整个闭包放进 `spawn_blocking`。

### 1.3 DB 缺少所有性能 PRAGMA

**文件**: `desktop/src-tauri/src/db.rs` (L7)
> `Connection::open` 后完全没有 WAL / busy_timeout / synchronous PRAGMA。默认 DELETE 日志每次写事务需要两次 fsync。
> **预期收益**: 写入吞吐提升 2-5×。

**建议**: 在 `init_database` 添加：
```rust
conn.execute_batch(
    "PRAGMA journal_mode=WAL;
     PRAGMA busy_timeout=5000;
     PRAGMA synchronous=NORMAL;
     PRAGMA foreign_keys=ON;"
)?;
```

### 1.4 load_config：22 次独立 SELECT（N+1 查询）

**文件**: `desktop/src-tauri/src/config.rs` (L79-153)
> 每个配置字段一次 `SELECT value FROM config WHERE key = ?`——共 22 次往返 + 22 次 Mutex 锁。

**建议**: 改为单次 `SELECT key, value FROM config` 批量查询，一次性解析所有字段（约 30 行改动）。

### 1.5 axum 缺少连接并发限制

**文件**: `desktop/src-tauri/src/mcp/http.rs` (L34-44)
> 无 `ConcurrencyLimitLayer`，恶意/失控客户端可无限创建并发连接耗尽 tokio 工作线程。

**建议**: 加 `tower::limit::ConcurrencyLimitLayer::new(256)`（1 行）。

### 1.6 每次审计写入都重新 open 文件

**文件**: `desktop/src-tauri/src/audit.rs` (L39-53)
> `write_audit_log` 每次调用都 `OpenOptions::open`（一次 syscall）。QPS 1000 = 1000 次 open/s。

**建议**: 在 AppState 维护共享 `BufWriter<File>`，定期 flush（~20 行改动）。

### 1.7 read_files 行范围读取加载整文件到内存

**文件**: `desktop/src-tauri/src/mcp/tools/read_files.rs` (L94-118)
> 读 100MB 日志的 `startLine=99000, endLine=99100` 时把 100MB 全部加载到内存。

**建议**: 用 `BufRead::lines()` 流式读取，跳前 N-1 行，只读需要的行数。

### 1.8 edit_files content.matches().count() 全量扫描

**文件**: `desktop/src-tauri/src/mcp/tools/edit_files.rs` (L102)
> `replace_all=false` 时只需判断 match_count 是 0/1/>1，但 `.count()` 扫描整个文件。

**建议**: `.matches(&f.old_string).take(2).count()`——早停在 >1 时（2 字符改动）。

## 二、P1 高优先——锁竞争 / 重复计算 / 事务缺失

### 2.1 auth_middleware 每请求锁 config 两次

**文件**: `desktop/src-tauri/src/mcp/http.rs` (L122-150)
> `config.read().await` 两次（token + 限流参数），可合并为一次。

**建议**: 一次 read 锁内取出 token + max_req + window_ms + rate_limit_enabled。

### 2.2 analyze_file 每次调用编译正则表达式

**文件**: `desktop/src-tauri/src/mcp/tools/analyze_file.rs` (L152-161)
> `regex::Regex::new(p).ok()` 每次请求重新编译。函数模式从不变化。

**建议**: 使用 `OnceLock<Regex>` 预编译所有语言函数模式（约 15 行）。

### 2.3 batch 审计写入使用同步阻塞调用

**文件**: `desktop/src-tauri/src/mcp/tools/batch.rs` (L88)
> `audit::write_audit_log` 同步调用在 async 循环中阻塞工作线程。10 个子操作 = 10 次阻塞写。

**建议**: 统一用 `spawn_blocking` 或收集后批量写入。

### 2.4 ensure_defaults 18 次独立 INSERT OR IGNORE（无事务）

**文件**: `desktop/src-tauri/src/db.rs` (L79-173)
> 每行 INSERT 在 DELETE 模式下是一次隐式事务 + fsync。

**建议**: 用 `BEGIN/COMMIT` 包裹（2 行）。

### 2.5 save_full_config 22 次独立 INSERT（无事务）

**文件**: `desktop/src-tauri/src/config.rs` (L166-254)
> 同理，22 次 `INSERT OR REPLACE` 在独立隐式事务中。

**建议**: 用 `BEGIN/COMMIT` 包裹（2 行）。

### 2.6 list_directory 每条目双重 stat 系统调用

**文件**: `desktop/src-tauri/src/mcp/tools/list_directory.rs` (L85-96)
> `read_dir` 返回的 DirEntry 已含文件类型和大小，额外调 `tokio::fs::metadata` 是多余 stat。50k 文件目录就是 50k 额外 syscall。

**建议**: `entry.metadata().await` 代替 `tokio::fs::metadata(&full_path).await`（1 行）。

### 2.7 write_audit_for_call 中序列化两次

**文件**: `desktop/src-tauri/src/mcp/http.rs` (L357-388)
> O1 的 auditMs 测量导致每请求 JSON 序列化两次。

**建议**: 用近似测量替代——序列化一次直接计时（无需额外 probe）。

### 2.8 前端 staleTime 为 0（每轮询强制不缓存）

**文件**: `desktop/src/main.tsx` (L8)
> `QueryClient` 未设置 `staleTime`，5s 轮询永不复用缓存。

**建议**: `staleTime: 3000`——在 3s 内多次 refetch 复用已有数据（1 行改动）。

### 2.9 前端无 React.memo（所有组件 5s 全树 diff）

**文件**: 全部组件
> `React.memo` 使用次数为 0。status 每 5s 变化 → 整棵虚拟 DOM 树重渲染。

**建议**: Header / ConectTab / SecurityTab / LogTab / SettingsTab 等中型组件加 `React.memo`（每个组件 1 行）。

### 2.10 securityBadges 数组每帧重建

**文件**: `desktop/src/components/layout/Header.tsx` (L35-49)
> 4 个 badge 对象 + JSX filter/map 每次渲染全新分配。

**建议**: 提为 `useMemo`（status 依赖）。

### 2.11 CommandPalette.items 每帧重建

**文件**: `desktop/src/components/layout/CommandPalette.tsx` (L107)
> 11 个 CommandItem 对象的数组每次渲染全新分配。

**建议**: 提为静态常量（不含动态数据）或 `useMemo`。

### 2.12 search_files Content 模式双重读文件

**文件**: `desktop/src-tauri/src/mcp/tools/search_files.rs` (L415-462)
> mmap 扫描后再次 `fs::read` 全文件到内存取上下文行。

**建议**: 从 mmap 结果直接提取上下文行（复用已映射内存）。

## 三、P2 中优先——内存 / 序列化 / 构建 / 渲染微优化

### 3.1 rate_limiter DashMap 无 GC（内存泄漏）

**文件**: `desktop/src-tauri/src/mcp/http.rs` (L158-168)
> 每个见过的 IP 永久留在 DashMap 中。长期运行随 IP 数量无界增长。

**建议**: 记录 last_cleanup，timestamps 为空时定期 remove（类似 gc_path_locks）。

### 3.2 read_recent_entries 整文件读入内存

**文件**: `desktop/src-tauri/src/audit.rs` (L55-77)
> 对所有行 `.collect()` 后才 `.rev().take(limit)`。

**建议**: 使用 `rev_lines` crate 从尾部反向读，或 `VecDeque` 固定容量。

### 3.3 write_files 自定义 base64 解码 O(64) 每字符

**文件**: `desktop/src-tauri/src/mcp/tools/write_files.rs` (L122-150)
> `table.iter().position()` 对每个字节线性搜索。

**建议**: 128 字节查找表实现 O(1) 解码（约 15 行，`LazyLock<[u8; 128]>`）。

### 3.4 cleanup_old_entries 同步阻塞启动路径

**文件**: `desktop/src-tauri/src/main.rs` (L153)
> `setup` 闭包中同步执行读 + 滤 + 写整个 audit.log，大文件阻塞窗口显示。

**建议**: 移到 `tauri::async_runtime::spawn` 后台任务。

### 3.5 load_config 部分加载失败静默回退

**文件**: `desktop/src-tauri/src/config.rs` (L82-149)
> `.unwrap_or_default()` 不记录日志。存储损坏时配置静默丢失。

**建议**: 至少 `log::warn!` 每个失败的字段。

### 3.6 vite.config.ts 无代码分割

**文件**: `desktop/vite.config.ts`
> 所有代码单 chunk。React + react-query + Tauri 全在一个 bundle。

**建议**: `manualChunks` 拆 vendor-react / vendor-query / vendor-tauri。

### 3.7 AboutGroup 大量内联 style 对象

**文件**: `desktop/src/components/tabs/AboutGroup.tsx` (L77/123/139-177/197)
> 每次渲染创建约 15 个新 style 对象。

**建议**: CSS 类替代或 `useMemo` 提取。

### 3.8 ConnectTab 计算值未 memo

**文件**: `desktop/src/components/tabs/ConnectTab.tsx` (L49-62)
> `displayHost` / `baseCommand` / `connectCommand` 每次渲染重算。

**建议**: `useMemo` 包裹，依赖 status/selectedIp。

### 3.9 Header status-pill 内联三元表达式过长

**文件**: `desktop/src/components/layout/Header.tsx` (L73-83)
> 启动失败/运行中/已停止三态的 className 三元链 inline 在 JSX 中，可读性差。

**建议**: 用 `useMemo` 提取 `statusPill` 对象。

### 3.10 深色主题 popover 未跟随 card 提亮

**文件**: `desktop/src/index.css` (L98-122)
> B6 调了 `--card` 但 `--popover` 保持 10%，弹出层比卡片暗。

**建议**: `--popover` 也调到 12% 与 `--card` 一致。

### 3.11 /health 端点无速率限制

**文件**: `desktop/src-tauri/src/mcp/http.rs` (L118-120)
> 攻击者可零成本高频调 /health DDoS。影响不大但应记录。

### 3.12 opt-level = "s" 偏向体积而非速度

**文件**: `desktop/src-tauri/Cargo.toml` (L101)
> 对 CPU 密集型操作（search_files 的正则匹配）非最优。但对网络 I/O 型服务合理。

**建议**: 未来针对性 `[profile.release.package."grep-searcher"]` override `opt-level = 3`。

### 3.13 edit_files 缺少 O1-b 的 replace_all 早停已有

已在 P0-1.8 中找到 .take(2) 早停方案。

## 四、P3 低优先——体积 / 启动 / 外观

### 4.1 png crate 仅用于解析 icon.png（可移除）

**文件**: `desktop/src-tauri/Cargo.toml` (L71)
> `png = "0.17"` 仅在启动期解码一次托盘图标。可改为构建时预渲染 RGBA 数组 `include_bytes!`，彻底移除 png 依赖（减小编译时间 + 二进制）。

### 4.2 create_dir_all 同步调用（启动期一次性）

**文件**: `desktop/src-tauri/src/main.rs` (L144)
> `std::fs::create_dir_all` 在 setup 闭包中同步执行。首次创建空目录开销可忽略。

### 4.3 网络变化检测 15s 固定轮询

**文件**: `desktop/src-tauri/src/main.rs` (L329)
> `tokio::time::sleep(Duration::from_secs(15))` 固定间隔。非瓶颈。

### 4.4 CSS 中硬编码颜色 30+ 处

**文件**: `desktop/src/index.css`
> `.c-front` / `.c-back` / `.s-row` 等组件样式有硬编码颜色，可选改用 CSS 变量。

### 4.5 db Mutex 单连接（低并发无影响）

**文件**: `desktop/src-tauri/src/state.rs` (L60)
> 这个单客户端 MCP server 场景单连接 + Mutex 完全够用。加 WAL 后读写可并发但 Mutex 阻止——不过 config 读写极少。

### 4.6 copy_files / move_files 纯顺序处理无并行

**文件**: `desktop/src-tauri/src/mcp/tools/copy_files.rs` (L25)
> 批量移动/复制 100 文件纯顺序。可用有限并发（但有 path_locks 死锁风险需小心）。

## 五、执行建议

**收益最大（改动小、效果显著）**：

| 优先级 | 条目 | 改动行数 | 预期收益 |
|--------|------|---------|---------|
| P0 | DB PRAGMA (WAL+busy+synchronous) | 4 行 | 写吞吐 2-5× |
| P0 | load_config N+1→单查询 | ~30 行 | 启动 DB 耗时 -90% |
| P0 | notebook_edit 同步→tokio::fs | 2 行 | 消除工作线程阻塞 |
| P0 | list_allowed_roots 同步→tokio | ~5 行 | 消除工作线程阻塞 |
| P1 | React.memo 包装 6 个中型组件 | 6 行 | 5s 轮询重渲染 -80% |
| P1 | QueryClient staleTime | 1 行 | 避免无效 round-trip |
| P1 | analyze_file 正则预编译 | ~15 行 | analyze_file CPU -50% |
| P0 | axum ConcurrencyLimitLayer | 1 行 | 防连接耗尽 |

> 建议先做「DB PRAGMA + 同步 I/O 修复 + React.memo + staleTime」这 6 条，总改动 < 50 行，覆盖最大收益。

## 六、未覆盖领域（建议后续补查）

- **Tauri IPC 序列化开销**：`invoke` 通过 JSON 序列化 `StatusResponse`（约 44 字段），每次 5s 轮询都序列化全量。考虑 split large/small status 两个端点减少不必要的字段传输。
- **内存 profile**：未做 heap allocation profiling（如 dhat/DHAT）。依赖 DashMap 多处使用，实际分配量需实测。
- **网络 RTT 影响**：LAN 环境低延迟（<5ms），所有优化都是微秒/毫秒级的，当前瓶颈不在网络本身。
