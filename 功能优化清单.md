## 定位与差距评估（2026-07-08）

**目标**：让远程 Linux 服务器上的 Claude Code 通过标准 MCP 协议直接读写本地 Windows 文件，最终对标 Claude Code 本地原生干活的**无差别体验**。

**现状**：文件搬运层已达 native 的 ~80%（读含行范围 / 列目录 / 名+内容搜索 / 删移复制 / 全量写 共 10 工具），但 native 的两大灵魂能力——**精准编辑（Edit）** 与 **命令执行（Bash）**——尚未具备。补齐 `edit_files` 可到 ~90%；Shell 是决定"无差别"上限的产品/安全抉择。

| 能力 | native | cc-bridge | 达标 |
|---|---|---|---|
| 读文件（行范围） | Read | `read_files` | ✅ |
| 列目录（递归） | — | `list_directory` | ✅ |
| 名/内容搜索 | Glob + Grep | `search_files` | ✅ |
| 删/移/复制 | — | `delete_files`/`move_files`/`copy_files` | ✅ |
| 写文件 | Write | `write_files`（仅全量覆盖） | ⚠️ |
| **精准编辑** | **Edit（串替换）** | 无 | ❌ P0 |
| **非 UTF-8 读取** | Read（自适应） | 仅 UTF-8（GBK 等失败） | ❌ P0 |
| **目录操作** | — | 无 mkdir/rmdir | ❌ P1 |
| **命令执行** | **Bash** | 无 | ❌ P2 |

## P 组 · 能力对齐（对标 native Claude Code）

### P0-1. 新增 `edit_files` 精准字符串替换 ✅ 【性价比最高】
- **现状**：只有 `write_files` 全量覆盖。改一行也要把整份文件跨公网传上来再传回，token 成本高、易截断出错，无 native 那种"外科手术式改几行"的流畅感。
- **方案**：新增 MCP 工具 `edit_files`，入参 `path` + `oldString` + `newString`（+ 可选 `replaceAll`）。后端：读文件 → 校验 `oldString` **唯一匹配**（非唯一/未找到则报错并提示）→ 替换 → 写回。
- **安全**：走同一条 `resolve_safe_path`（白名单 + canonicalize），受只读模式拦截、写前 backup。**安全模型不变**。
- **涉及（工具三件套）**：`src/mcp/tools/edit_files.rs` 新建 handler → `tools/mod.rs` 声明 `pub mod` → `http.rs` `dispatch_tool` 加分支 + `get_tool_definitions` 加定义 + `WRITE_TOOLS` 数组加入 `edit_files`。README「MCP 工具」表数量 +1。
- **收益**：立刻把体验从 ~80% 拉到 ~90%，这是投入产出比最高的一步。

### P0-2. 读取编码自适应（GBK 等非 UTF-8）✅
- **现状**：`read_files` 用 `read_to_string`（仅 UTF-8），非 UTF-8 文件（GBK/GB18030 等）直接读取失败。用户自身的 NC65 项目正是 GBK 文件，会踩坑。
- **方案**：读取时先探测编码（BOM / 启发式，考虑 `encoding_rs` 或 `chardetng`），非 UTF-8 转码为 UTF-8 返回；`analyze_file` 已有编码探测可复用。写回时保留原编码（可选）。
- **涉及**：`read_files.rs`（可能抽 `security/encoding` 或 `lib`），依赖 `encoding_rs`（注意二进制体积影响，评估后决定）。
- **收益**：中文 Windows 环境下真实可用，避免"读不了自家文件"。

### P1-1. 目录操作 `create_directory` / `remove_directory` ✅
- **现状**：`write_files` 会自动建父目录，但没有独立的 mkdir；`delete_files` 明确不删目录，缺 rmdir。
- **方案**：新增 `create_directory`（幂等 mkdir -p）与 `remove_directory`（可选递归，递归需二次确认语义或参数显式声明）。
- **安全**：同白名单 + 只读拦截；`remove_directory` 递归删除风险高，需在描述中警示并纳入 `WRITE_TOOLS`。
- **涉及**：工具三件套 + README。
- **收益**：补齐目录级操作，减少"只能靠写文件间接建目录"的别扭。

### P2. 命令执行（Shell）— 方案 3：完整 shell + 醒目风险开关 ⬜ 【已选定 · 需专项方案 + 二次确认】
- **背景**：native Claude Code 战斗力一大半来自跑命令（测试 / git / build / lint / ripgrep）。缺 Shell 意味着远程 Claude 能写对代码但**无法自我验证**，是与 native 最大的体验断层。
- **决策（用户已选）**：走**方案 3 —— 完整 shell 执行**，配醒目风险开关，追求真正无差别体验。
- **重大安全影响（必须在实现方案与 UI 中如实呈现）**：
  - 暴露完整 shell = 授予远程服务器**任意代码执行权（RCE）**。
  - 一旦开启，路径白名单 / 只读模式 / 扩展名限制等现有约束**可被 `bash -c` 等绕过**——安全模型发生**质变**。
  - Bearer token 鉴权 + 限流仍是唯一的准入防线，绝不可削弱。
- **拟定护栏（待专项方案细化）**：
  1. **默认关闭**，设置页独立"命令执行"开关，开启时强制**醒目红色风险确认弹窗**（明确告知 RCE 含义）。
  2. 开关状态在顶栏 / 安全页有**持久醒目徽章**（类比白名单关闭时的红警示）。
  3. 新增 MCP 工具 `run_command`（cwd 限制在白名单根内、超时、输出大小上限、stdout/stderr 分离、退出码回传）。
  4. 每条命令**必审计**（即使审计开关关闭也应记录命令执行，或至少不允许静默）。
  5. 评估 stdin / 交互式命令、长任务流式输出、进程终止能力。
  6. `run_command` 纳入"写操作"语义：只读模式开启时应拒绝（或至少拒绝非只读命令——需权衡实现复杂度）。
- **涉及**：新工具三件套 + 新 config 开关（`shell_enabled` 默认 false）+ 前端开关卡 + 风险确认弹窗 + 顶栏/安全页徽章 + README + CLAUDE.md 安全约束章节补充。
- **状态**：方向已定，**实现前需单独出完整方案（含安全评审）再动手**，不在本轮小步范围内。
- **收益**：跨过此步，远程体验才真正逼近 native "无差别"。

## 体验优化（UI/交互，不改后端安全约束）

## A 组 · 低成本高收益（推荐先做）

### A1. Toast 反馈统一 ⬜
- **现状**：复制 / 保存 / 重生 token 的成功反馈是内联小字 "✓"，容易被漏看。
- **方案**：改成右下角浮层 toast（引入 `sonner`，约 +3KB）；离线 fallback 用自写轻量 toast。
- **涉及**：`App.tsx` 挂 `<Toaster>`；各处复制/保存回调改调 `toast.success()`。
- **收益**：反馈显眼、统一，专业感提升。

### A2. Token 一键复制 + 眼睛切换 ⬜
- **现状**：连接页 Bearer token 需手动选中复制，且明文常驻。
- **方案**：加复制按钮（Copy/Check 图标切换）+ 显示/隐藏切换（Eye/EyeOff）。
- **涉及**：`ConnectTab.tsx` token 展示区。
- **收益**：安全（默认隐藏）+ 便捷。

### A3. 端口占用提示 ⬜
- **现状**：MCP 启动失败（端口被占）时，顶栏静默停在"已停止"，无原因。
- **方案**：启动失败时状态胶囊标红并给出原因（端口占用 / 绑定失败）。
- **涉及**：`mcp/http.rs` 绑定失败回传错误 → `StatusResponse` 增错误字段 → `Header.tsx` 展示。
- **收益**：故障可诊断，减少困惑。

## B 组 · 中等成本

### B4. 首次启动引导空态 ⬜
- **现状**：白名单为空时，安全页只显示空列表，新用户不知道下一步。
- **方案**：空态引导卡（图标 + "点此添加第一个可访问目录" + 按钮）。
- **涉及**：`SecurityTab.tsx` / `ConnectTab.tsx` 空态分支。
- **收益**：新用户上手更顺。

### B5. 图标线性化 ⬜
- **现状**：部分位置仍用 emoji 图标，风格不统一。
- **方案**：统一换成线性 SVG（现有 `ui/icon.tsx` 组件可扩，零依赖）；或引入 `lucide-react`。
- **涉及**：全局图标点位。
- **收益**：视觉专业度提升。

### B6. 深色主题层次微调 ⬜
- **现状**：深色下卡片 / 边框 / muted 文字对比略弱，层次不够清晰。
- **方案**：对齐标杆 midnight 主题，调 `--card-bg` / `--border` / `--muted` 令牌。
- **涉及**：`index.css` 深色变量。
- **收益**：深色可读性。

---

## C 组 · 可选

### C7. 审计日志搜索 / 过滤 ⬜
- **现状**：日志条目多了以后无法快速定位。
- **方案**：按工具名 / 时间 / 关键字过滤。
- **涉及**：`LogTab.tsx` 增过滤 UI + 前端过滤逻辑。
- **收益**：大日志量下的可用性。

### C8. 配置导出 / 导入 ⬜
- **现状**：换机 / 重装需手动重配白名单和开关。
- **方案**：一键导出（白名单 + 5 个开关 + 网络配置）为 JSON，另一端导入。
- **涉及**：新增 `export_config` / `import_config` 命令 + 设置页按钮。
- **收益**：迁移便捷。
