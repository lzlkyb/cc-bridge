### P0-1. 新增 `edit_files` 精准字符串替换 ✅ 【性价比最高】
- **现状**：只有 `write_files` 全量覆盖。改一行也要把整份文件跨公网传上来再传回，token 成本高、易截断出错，无 native 那种"外科手术式改几行"的流畅感。

- **安全**：走同一条 `resolve_safe_path`（白名单 + canonicalize），受只读模式拦截、写前 backup。**安全模型不变**。
- **涉及（工具三件套）**：`src/mcp/tools/edit_files.rs` 新建 handler → `tools/mod.rs` 声明 `pub mod` → `http.rs` `dispatch_tool` 加分支 + `get_tool_definitions` 加定义 + `WRITE_TOOLS` 数组加入 `edit_files`。README「MCP 工具」表数量 +1。
- **收益**：立刻把体验从 ~80% 拉到 ~90%，这是投入产出比最高的一步。

### P0-2. 读取编码自适应（GBK 等非 UTF-8）✅
- **现状**：`read_files` 用 `read_to_string`（仅 UTF-8），非 UTF-8 文件（GBK/GB18030 等）直接读取失败。用户自身的 NC65 项目正是 GBK 文件，会踩坑。

- **涉及**：`read_files.rs`（可能抽 `security/encoding` 或 `lib`），依赖 `encoding_rs`（注意二进制体积影响，评估后决定）。
- **收益**：中文 Windows 环境下真实可用，避免"读不了自家文件"。

### P-候选. run_command 改用 portable-pty（ConPTY）修复真实子进程输出丢失 ✅【已根原因 + 回退 CREATE_NO_WINDOW，2026-07-10】
- **发现的 bug**：`run_command` 无论前台/后台，只要命令是一个真实独立的 .exe（`hostname.exe`/`git.exe`/`cargo.exe`——不是 `echo`/`type` 这类 cmd 内置命令），stdout/stderr 都读不到任何内容（但 exitCode 一直正确）。根因推测：外层 cmd.exe 用 `DETACHED_PROCESS`（完全无控制台）生成，它再往下 spawn 真实 exe（孙进程）时，Windows 对这种无控制台进程的控制台子系统孙进程处理不干净。
- **候选方案**：改用 `portable-pty = "0.9"` 的 ConPTY 代替 `Stdio::piped()+DETACHED_PROCESS`，保留现有 win32job 集成（`Child::as_raw_handle()` 直接嗂给 `process_job::create_and_assign()`）。
- **取舍（用户确认）**：换回 `CREATE_NO_WINDOW` 能修好输出丢失，但会重新暴露已知的 MSVC 并发链接崩溃（`STATUS_CONTROL_C_EXIT`），用户选择不接受这个取舍，改为写 ConPTY 候选方案。
- **已知行为变化**：PTY 天然只有一路输出，stdout/stderr 合并，`stderr` 字段固定为空。
- **进展（2026-07-10 已收口）**：本地实测确认 `portable-pty` 方案**根本性不可用**——`cmd.exe` 启动后会发 DSR 查询 `\x1b[6n` 并等待终端应答，而 portable-pty 0.9 的 `MasterPty`/`SlavePty` **未实现 `Write`**，无法应答，导致 `cmd` 永久挂起（stdout 空、exitCode 超时）。5/5 候选测试全部失败，验证后判定该路线在 Windows 上走不通。
- **最终决策（用户选回退 CREATE_NO_WINDOW）**：彻底回退 `portable-pty`，改用 `CREATE_NO_WINDOW (0x08000000) | CREATE_NEW_PROCESS_GROUP (0x00000200)` + `Stdio::piped()` 直接 spawn `cmd /C`。stdout/stderr 各自独立管道，真实 `.exe` 子进程（hostname/git/cargo）输出**现在能正确捕获**且两路分离；`Cargo.toml` 已移除 `portable-pty` 依赖。实测 `cargo build` 在 `CREATE_NO_WINDOW` 下**未复现**历史 `STATUS_CONTROL_C_EXIT` 崩溃（与 P2 早期笔记里"DETACHED_PROCESS 才不崩"的判断相反——当前 toolchain 下 CREATE_NO_WINDOW 稳定）。
- **验证**：`run_command` 单元覆盖从 0 → 9（含 `foreground_real_exe_returns_stdout` 直接复现原 bug），`cargo test --lib` 全绿（39 passed/3 ignored）、`cargo clippy` 零警告；版本号同步递增至 2.2.14。

### P4-1. 进程树终止改用 `win32job`（替换三处重复的 taskkill 手写调用，共同修 D18）✅ 【已完成 2026-07-10】
- **现状**：`run_command.rs` 和 `stop_command.rs` 各自手写了一份几乎相同的 `taskkill /T /F /PID` 调用（对应清单 D18 记录的重复代码）。taskkill 还有个本质缺陷：它是按调用时看到的 PID 快照去杀，子孙进程刚 fork 出来还没被系统记录全时可能漏杀；cc-bridge 自身意外崩溃/被杀时，后台命令的子孙进程会变成孤儿，永远无人清理。
- **方案**：改用 **`win32job`**（[crates.io](https://crates.io/crates/win32job)，最新 2.0.3，103 万次下载，安全包装 Windows Job Object 的 API，VS Code/Chrome/Docker Desktop 在 Windows 上管理子进程都用这个系统原语）。每次 `run_command` spawn 时把子进程挂进一个 Job Object，设置 `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`：Job 跟踪所有曾挂在它下面的进程（不管嵌套几层），不依赖某一时刻的 PID 快照，天然没有 taskkill 的竞态问题；cc-bridge 主进程自己崩溃/被杀时，挂在 Job 下的后台命令会被系统自动清理，这是 taskkill 方案完全做不到的。
- **涉及**：实际实现时发现不是两处而是**三处**手写 taskkill（`run_command.rs`/`stop_command.rs`/`commands.rs::stop_running_command`），已新建 `process_job.rs` 封装 `create_and_assign()`，三处调用点统一改用：spawn 时立即用 `child.raw_handle()` 挂进 Job，`run_foreground` 超时/结束时 `drop(job)`，`stop_command`/`stop_running_command` 改为移除 `RunningCommand` 注册项后 drop 即可，不再调 taskkill.exe。`RunningCommand` 结构体新增 `job: win32job::Job` 字段持有它直到被移除。
- **行为变化（有意为之）**：前台命令正常结束后也会 drop job，顺带杀掉可能残留的 detached 子孙进程（如 `cmd /C` 里用 `start` 开出去的后台进程）——以前这类进程会在调用返回后继续存活，现在会跟着一起终止。
- **安全**：仍受 `shell_enabled`/`readonly_mode` 约束，不放宽任何现有护栏，只是把进程终止机制换成更可靠的系统原语。
- **验证**：版本号同步递增至 2.2.9（`Cargo.toml` + `tauri.conf.json`），远程 `cargo check` 通过（exit code 0），用户确认无问题。建议后续跟一次 `cargo clippy` + 实际跑一个 `run_command`/`stop_command` 端到端验证进程树真能被杀干净。
- **收益**：消除孤儿进程风险 + 顺带修掉 D18 的代码重复（三处而不是两处）。
- **迁移收尾（2026-07-10, v2.2.16）**：手写 `win32job` 已进一步替换为社区维护的 **`process-wrap`**（`=8.0.2`，watchexec 出品，`command-group` 官方后继，CI 覆盖 Windows/Linux/macOS）。删除 `process_job.rs`（111 行 Win32 FFI）及其 `#[ignore]` 自伤测试。`process-wrap` 的 `JobObject` 内部先以 `CREATE_SUSPENDED` 启动再 resume，**消除了原"先 spawn 再 assign"的孙进程漏杀竞态窗口**（比手写 win32job 更正确）。终止语义变化：`process-wrap` std `JobObject` 默认**不** kill-on-close，故 `stop_command`/超时分支改为显式 `child.start_kill()`（`TerminateJobObject`）；`RunningCommand.job` 改为 `child: Arc<Mutex<Box<dyn StdChildWrapper>>>`。版本锁 8.0.2 理由：对齐 windows 0.56 避免第二份绑定 + 必须开 `tracing` feature（否则 `job_object` 模块无条件调 `debug!` 编译失败）。验证：`cargo test --lib` 41 passed/0 ignored、`clippy` 零警告。

### P4-2. 目录遍历/通配符匹配改用 `ignore` + `globset`（ripgrep 同款）✅ 【已完成 2026-07-10，仅限 search_files】
- **现状**：`search_files.rs`/`list_directory.rs` 是手写递归异步 walker（`tokio::fs::read_dir` + boxed future 递归），**没有任何 `.git`/`node_modules`/`target` 排除逻辑**，远程搜一个真实 Node/Rust 项目会真的递归进 `node_modules` 里几万个文件，又慢又污染搜索结果。同文件里的 `glob_to_regex` 手写通配符转换，**只支持 `*` 和 `?`**，不支持 `**`、`[abc]` 字符类、`{a,b}` 花括号。
- **方案**：改用 **`ignore`**（[crates.io](https://crates.io/crates/ignore)，最新 0.4.28，**1.44 亿次下载**，ripgrep 作者 BurntSushi 维护，自带 `.gitignore` 感知的目录遍历，默认跳过 `.git`）替换两处手写 walker；用 **`globset`**（[crates.io](https://crates.io/crates/globset)，最新 0.4.18，**1.97 亿次下载**，同样是 BurntSushi 维护，`ignore` 内部依赖它）替换 `glob_to_regex`，获得完整 glob 语义。
- **实际范围（比原计划窄）**：只改了 `search_files.rs`。`list_directory.rs` 这次没动——`ignore::Walk` 是扁平迭代器，与 `list_directory` 现有的嵌套树输出契约（`children: Option<Vec<DirEntry>>`）不契合，重建树需要额外复杂度；且它只列名字不读内容，性能代价远低于 `search_files`。留待真有人反馈再单独处理。
- **涉及**：`Cargo.toml` 加 `ignore = "0.4"` + `globset = "0.4"`；`search_files.rs` 手写 `glob_to_regex`（只认 `*`/`?`）换成 `globset::GlobBuilder`（保留原来的大小写不敏感）；手写递归 walker 换成 `ignore::WalkBuilder`。因为 `ignore::Walk` 是同步/阻塞迭代器，整个遍历+匹配逻辑丢进 `tokio::task::spawn_blocking`，内部改用 `std::fs::read_to_string`（不再混用 async fs）。输出 JSON 结构（`matches: [{path, type, lineNumber, ...}]`）保持不变。
- **踩到的坑（已修正）**：本来想用 `hidden(false)` 保留现有"不隐藏点文件"行为，但实测/查证发现 `hidden(false)` + 默认 `git_ignore(true)` 会让 `.git` 目录本身也被遍历进去（ripgrep 自己的已知行为，[BurntSushi/ripgrep#3099](https://github.com/BurntSushi/ripgrep/issues/3099)，官方 wontfix）。所以额外用 `ignore::overrides::OverrideBuilder::new(root).add("!.git")` 强制排除，不依赖 hidden 开关。
- **不加开关**：用户确认不需要 `respectGitignore` 这类可选参数，默认就遵守 `.gitignore`，与 native Claude Code 本地 Grep/Glob 工具的默认行为保持一致。
- **安全**：仍走现有 `resolve_safe_path` 白名单校验，不引入新的路径解析逻辑；被 `.gitignore`/`!.git` 跳过只是这次搜不到，不影响直接 `read_files` 指定路径读取。
- **验证**：版本号同步递增至 2.2.11，远程 `cargo check` 通过（exit code 0）。
- **收益**：`search_files` 内容搜索不再白白读遍 `node_modules`/`target`（前提是目标项目有干净的 `.gitignore`）；`namePattern` 支持完整 glob 语义（`**`、`[abc]` 等），无条件生效。

### P6-1. `search_files` 并行遍历（对标 ripgrep 速度，吃掉服务端 75% 耗时）✅ 【已完成，2026-07-10 真实审计数据驱动】
- **背景（2026-07-10 实测）**：用 `tools/audit_stats.py` 统计本机真实 `audit.log`（路径 `Roaming/com.ccbridge.desktop/audit.log`，315 条有效记录、服务端总墙钟 72.8s），发现 **`search_files` 独占 54.3s = 74.6%** 的服务端耗时，平均 1698ms、p95=9391ms、最坏 19859ms；而 edit_files 仅 2.1%、read_files 0.4%、write_files 0.0%（读改写本身极快，平均 3–21ms）。即"比 claude code 慢"的体感主因不是改文件慢，而是**搜索慢**——本地 Claude Code 用并行 ripgrep 约 50–200ms 返回，这里每次 1–20s。这也**纠正了早期"审计/备份同步写是主因（D7）"的判断**：那两条只影响已占 ~3% 且很快的读写改路径，ROI 极低，其相对优先级已降级（见 D7 末注）。
- **现状（已读 `search_files.rs`）**：P4-2 把手写 walker 换成了 `ignore::WalkBuilder`，但用的是 **`build()` 单线程串行**遍历整棵树（ripgrep 快是因为 `build_parallel()` 多线程）；内容搜索还把每个文件 `read_to_string` 整读、再把所有行 `collect` 成 `Vec`，大文件/大目录开销明显。当搜索根恰好没有干净 `.gitignore`（如本项目 `desktop/` 直接搜、或撞进 `target/` 千级 `rlib`/`rmeta`）时，整棵树串行扫一遍就是秒级。
- **方案**：
  1. 遍历改 `ignore::WalkBuilder::build_parallel()`，交给 `ignore` 内部的 rayon 线程池多线程走树——这是逼近 ripgrep 速度的关键。
  2. **强制排除 `target/` `node_modules/` `.git/`，不依赖搜索根是否恰好有 `.gitignore`**（这是目前最坏 20s 尾巴的直接来源）。
  3. 跨线程共享原子 `AtomicUsize` 计数 `max_results`，任意线程达标即早停（配合 `ignore::WalkState::Quit` 让并行 walker 提前收尾）。
  4. 内容匹配走字节惰性扫描（`regex::bytes::Regex` 或按行 `BufRead` 流式），**不整读、不 `collect` 全行**；每行命中只推一次结构化结果。
- **涉及**：`mcp/tools/search_files.rs`（把 `walk_search_blocking` 重写为并行版；输出 JSON 结构 `matches: [{path,type,lineNumber,...}]` 保持不变）。仍在既有 `spawn_blocking` 内跑（并行 walk 本就吃满 CPU，不占 tokio reactor）。
- **安全**：仍走现有 `resolve_safe_path` 白名单校验，不引入新路径解析逻辑；强制排除只是这次搜不到，不影响 `read_files` 指定路径直读。
- **验证**：改完用 `tools/audit_stats.py` 复跑真实 `audit.log`，重点看 `search_files` 的 avg/p95 是否降到 100–300ms 量级、`search_files` 总占比是否从 74.6% 跌到个位数；可造含 `target/` 假目录的 fixture 单测验证强制排除生效。
- **收益**：直接抹掉约 75% 的服务端墙钟，是"比 claude code 慢很多"最直接的体感修复；预计 5–20×。
- **2026-07-10 实施记录**：`walk_search_blocking` 已重写为并行版——`build_parallel()` + `Mutex<Vec>` 收集 + 原子 `head_limit` 早停（`WalkState::Quit`）；`OverrideBuilder` 强制排除 `!.git/!target/!node_modules`（不依赖 .gitignore）；`files_with_matches`/`count` 两模式改 `BufRead` 惰性流式（命中即停/只数不存），`content` 模式保留 `read_to_string`（受 `max_file_size` + 强制排除约束）；并行结果按 `path`(+行号) 排序保证确定性。新增单测 `force_excludes_build_dirs_without_gitignore` 固化强制排除；原有 7 个单测全部保留（因排序改为确定性，依赖顺序的断言已安全）。`cargo build --lib` 通过，`cargo test --lib` 全绿（8 passed，含新增 `force_excludes_build_dirs_without_gitignore`）；`cargo fmt` 通过。
